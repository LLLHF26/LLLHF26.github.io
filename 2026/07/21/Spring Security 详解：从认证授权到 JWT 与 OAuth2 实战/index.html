Spring Security 详解：从认证授权到 JWT 与 OAuth2 实战

一、认证 vs 授权：一字之差，天壤之别

很多开发者把「认证」和「授权」混为一谈，这是安全领域最常见也最危险的认知误区。

维度 认证（Authentication） 授权（Authorization）
回答的问题 你是谁？ 你能做什么？
英文 Authentication（AuthN） Authorization（AuthZ）
生活类比 刷身份证进大楼 刷工卡进特定楼层
技术手段 用户名密码、手机验证码、指纹、OAuth2 登录 RBAC 角色权限、ABAC 属性权限、ACL 访问控制
失败后果 陌生人进入系统 合法用户越权操作
Spring Security 组件 AuthenticationManager → AuthenticationProvider → UserDetailsService AuthorizationManager → AccessDecisionManager（已废弃）

一句话总结：认证解决「你是谁」，授权解决「你能干什么」。先证明身份，再判断权限，缺一不可。

二、Spring Security 核心架构全解

1、整体架构鸟瞰

Spring Security 的本质是一组 Servlet Filter 的链式调用，在请求到达 Controller 之前完成认证和授权。

2、核心对象关系

理解以下五个核心对象及其关系，是掌握 Spring Security 的关键：

核心对象 职责 常见实现
AuthenticationManager 认证入口，委托给 Provider ProviderManager
AuthenticationProvider 执行具体认证逻辑（密码比对等） DaoAuthenticationProvider
UserDetailsService 按用户名加载用户信息 InMemoryUserDetailsManager / JdbcUserDetailsManager / 自定义
UserDetails 封装用户名、密码、权限、账户状态 User（内置）/ 自定义实体
PasswordEncoder 密码加密与验证 BCryptPasswordEncoder（推荐）
SecurityContext 持有 Authentication 对象 SecurityContextImpl
SecurityContextHolder 全局访问 SecurityContext（ThreadLocal） 默认 ThreadLocal 策略
AuthorizationManager 授权决策（7.0 起统一） AuthorityAuthorizationManager / 自定义

3、SecurityContextHolder：安全上下文的”记忆中枢”

SecurityContextHolder 是整个安全体系的”记忆中枢”——它在请求生命周期内保存当前用户的身份信息，后续所有授权判断都依赖它。

三种策略：默认 MODE_THREADLOCAL（线程隔离），还有 MODE_INHERITABLETHREADLOCAL（子线程继承）和 MODE_GLOBAL（全局共享）。Web 应用几乎总是用默认策略。

三、认证实战：从表单登录到 JWT

1、最小化安全配置（Spring Security 7.x 风格）

Spring Security 7.0 重大变更：Lambda DSL（Customizer lambda）成为唯一支持的配置方式，旧式 .and() 链式调用被彻底移除。每个 configurer 接收一个 Customizer lambda，不再有 .and() 跳转。

2、密码加密：为什么绝对不能存明文？

方案 安全性 说明
明文存储 ❌ 极危险 数据库泄露=全军覆没
MD5 / SHA-1 ❌ 已破解 彩虹表秒破
MD5 + Salt ⚠️ 不够 可被暴力破解
BCrypt ✅ 推荐 自带随机盐，可调 cost（默认 10）
Argon2 ✅ 最强 2015 密码哈希竞赛冠军，抗 GPU
PBKDF2 ✅ 可用 NIST 标准，迭代次数需调高

BCrypt 关键参数：strength（即 cost factor）默认 10，每次哈希约 100ms。每增加 1，耗时翻倍。生产环境建议 10-12。

4、JWT 无状态认证：前后端分离标配

JWT（JSON Web Token）是前后端分离架构下最主流的认证方案。核心思路：登录成功后签发 Token，后续请求携带 Token，服务端无状态验证。

JWT 的三大痛点与解法：

  • 无法主动失效：Token 签发后在过期前一直有效 → 用 Redis 维护黑名单，退出登录时将 Token 加入黑名单
  • 续签问题：Token 过期后需要重新登录 → 实现 Refresh Token 机制，用 Refresh Token 换取新的 Access Token
  • Token 泄露：XSS 可偷 localStorage 中的 Token → 敏感操作要求二次认证，或使用 HttpOnly Cookie + CSRF Token

注解 时机 支持 SpEL 来源 推荐度

六、CSRF、CORS 与会话管理

1、CSRF 防护

CSRF（跨站请求伪造）攻击原理：攻击者诱导已登录用户在不知情的情况下发送恶意请求。

场景 CSRF 策略 原因
表单提交的传统 Web 应用 ✅ 开启 Cookie 自动携带，需要 CSRF Token 防护
SPA + Cookie 认证 ✅ 开启（SPA 模式） Cookie 仍会自动携带
SPA + JWT（Bearer Token） ❌ 关闭 Token 从 Header 手动携带，不依赖 Cookie

2、CORS 跨域配置

Spring Security 7.1 新增：PreFlightRequestFilter 专门处理 CORS 预检请求，不再需要预检请求穿过整个安全过滤器链，提升了性能。

3、会话管理

九、高频面试题

Q1：Spring Security 的认证流程是怎样的？

用户提交用户名密码 → UsernamePasswordAuthenticationFilter 拦截 → 创建 UsernamePasswordAuthenticationToken（未认证） → 交给 AuthenticationManager → 委托给 DaoAuthenticationProvider → 调用 UserDetailsService.loadUserByUsername() 查询用户 → PasswordEncoder.matches() 比对密码 → 成功则创建已认证的 Authentication 对象 → 存入 SecurityContextHolder → 失败抛出 BadCredentialsException。

Q2：JWT 和 Session 的区别？各自适用什么场景？

维度 Session JWT
状态 有状态（服务端存储） 无状态（Token 自包含）
扩展性 需要共享 Session（Redis） 天然支持水平扩展
主动失效 简单（删除 Session） 需要黑名单机制
续签 自动 需要 Refresh Token
安全性 CSRF 风险 XSS 风险
适用场景 传统 Web 应用 前后端分离 / 微服务

Q3：@PreAuthorize 和 @Secured 有什么区别？

@PreAuthorize 支持 SpEL 表达式，可以写复杂逻辑（如 hasRole('ADMIN') or #article.author == authentication.name），灵活性高。@Secured 只支持简单角色检查（@Secured("ROLE_ADMIN")），不支持 SpEL。生产环境推荐 @PreAuthorize。

Q4：Spring Security 7.0 为什么要移除 OAuth2 Password Grant？

Password Grant 要求前端直接传递用户名密码给客户端应用，存在三大安全风险：①客户端应用接触用户凭据，泄露风险高；②无法支持 MFA/SSO 等增强认证；③违反 OAuth2 设计初衷（资源所有者不直接与授权服务器交互）。7.0 强制使用 Authorization Code + PKCE，前端只拿到 Authorization Code，不接触用户密码，安全性大幅提升。

Q5：如何实现 RBAC（基于角色的访问控制）？

RBAC 三层模型：用户 → 角色 → 权限。在 Spring Security 中：

  1. UserDetails.getAuthorities() 返回 ROLE_ADMIN 和 article:read 形式的权限
  2. URL 级用 .hasRole("ADMIN") / .hasAuthority("article:write")
  3. 方法级用 @PreAuthorize("hasAuthority('article:write')")
  4. 数据库设计：user 表 → role 表 → permission 表，中间通过 user_role 和 role_permission 关联表连接

Q6：JWT 如何实现主动失效？

JWT 本身无状态，签发后无法主动撤销。常见方案：

  1. 黑名单：退出登录时将 Token 加入 Redis，TTL 设为 Token 剩余有效期。每次请求额外检查黑名单
  2. Refresh Token 轮换：Access Token 短期（如 15 分钟），Refresh Token 长期（如 7 天）。退出时删除 Refresh Token，Access Token 自然过期
  3. 版本号机制：用户表加 token_version 字段，JWT 中携带版本号，修改密码时版本号 +1，旧 Token 失效

Q7：Spring Security 7.0 中 PathPatternRequestMatcher 有什么变化？

7.0 统一使用 PathPatternRequestMatcher（基于 Spring 的 PathPatternParser），移除了 AntPathRequestMatcher 和 MvcRequestMatcher。关键变化：** 通配符只能出现在模式末尾，中间出现会抛出异常。如 /api/**/admin 不再合法，需改写为 /api/** 然后在代码中做二次校验。这提升了匹配性能和安全性。

十、总结与选型指南

场景 认证方案 会话策略 CSRF
传统 SSR Web 应用 表单登录 + Session IF_REQUIRED ✅ 开启
前后端分离 SPA JWT（Bearer Token） STATELESS ❌ 关闭
微服务间调用 OAuth2 Client Credentials 无 ❌
第三方登录 OAuth2 Authorization Code + PKCE 按需 视情况
企业级 SSO SAML 2.0 / OIDC 集中会话 ✅
高安全要求 MFA 多因子 + WebAuthn 按需 ✅

Spring Security 作为 Java 生态最成熟的安全框架，7.0 / 7.1 的演进方向非常清晰：消除不安全默认值、统一 API 风格、拥抱现代认证协议（WebAuthn/MFA/OAuth2 PKCE）。理解核心架构（FilterChain → Authentication → Authorization），掌握 JWT 和 OAuth2 两条主线，关注 7.x 的 Breaking Changes，就能在生产中做到安全架构能设计、代码能落地、面试能答、升级能避坑。

核心心法：安全没有银弹，只有纵深防御。认证是门，授权是锁，加密是墙，审计是眼。每一层都不能少。
