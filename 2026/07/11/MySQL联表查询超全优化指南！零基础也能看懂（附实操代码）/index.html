二、核心优化一：大表关联字段必建索引（最关键）

问题根源

联表查询的核心逻辑是：遍历驱动表数据，去被驱动表匹配数据。如果大表的关联字段没有索引，MySQL会触发块嵌套循环（BNL），全盘扫描数据，性能暴跌。

未优化（错误写法，全表扫描）

问题特征：执行计划出现 type=ALL、Using join buffer (Block Nested Loop)

优化方案：给大表关联字段加索引

优化效果

执行计划中 type 变为 ref，告别全表扫描，查询效率提升数十倍。

核心口诀：小表无所谓，大表关联字段必须加索引！

三、核心优化二：小表驱动大表，杜绝反向驱动

MySQL联表底层是嵌套循环查询：驱动表遍历行数越少，联表效率越高。

铁律：用数据量小的小表，驱动数据量大的大表

正确写法（小表驱动大表）

重点：LEFT JOIN 特殊规则

LEFT JOIN 左表固定是驱动表，绝对不能写反！

四、核心优化三：先过滤，再关联（大幅减少笛卡尔积）

很多人写SQL习惯「先关联整张表，再过滤数据」，这是超级大坑。

未优化时，会先完成全量联表，生成巨大的笛卡尔积，再筛选数据，资源消耗极高。

错误写法（先关联、后过滤）

正确写法（先缩小驱动表范围，再联表）

五、核心优化四：统一字段类型，杜绝隐式类型转换

这是最隐蔽的坑：关联字段类型不一致，会直接导致索引失效。

原理：MySQL会对索引列做隐式类型转换，相当于对索引列使用函数，索引直接报废，触发全表扫描。

模拟坑场景

失效SQL（索引完全没用）

解决方案

保证联表关联字段 类型、长度、字符集、是否可为NULL 完全一致！

六、终极优化：覆盖索引（彻底消除回表，性能拉满）

普通索引联表会存在回表查询，我们可以通过覆盖索引，让查询数据全部从索引中获取，无需回表，实现最优性能。

优化标识：EXPLAIN执行计划出现 Using index，代表触发覆盖索引，无回表，性能极致。

七、优化结果校验：如何判断是否优化成功？

执行 EXPLAIN 查看执行计划，对照以下标准：

  • ❌ type=ALL：全表扫描，需加索引

  • ❌Using join buffer (Block Nested Loop)：无索引联表，严重卡顿

  • ✅ type=ref：索引生效，联表正常

  • ✅ Using index：覆盖索引，无回表，最优状态

八、联表优化终极总结（直接背！）

  1. 大表关联字段必建索引，杜绝块嵌套循环全表扫描

  2. 小表驱动大表，LEFT JOIN 小表必须放左侧

  3. 先过滤、后关联，最大限度缩小驱动表数据量，减少笛卡尔积

  4. 关联字段类型完全一致，禁止隐式类型转换导致索引失效

  5. 高频查询场景，搭建覆盖索引，消除回表提升性能
