Java 并发编程详解：从 JMM 到 JUC 核心源码

一、并发编程基础：你真的理解线程吗？

很多人写多线程代码时直接 new Thread()，出了问题就加 synchronized。这种”摸着石头过河”的方式，在单测里可能没问题，但一到生产环境就原形毕露。并发编程的第一步，是把基础概念打牢。

1、线程 vs 进程：最本质的区别

维度 进程 (Process) 线程 (Thread)
资源分配 操作系统资源分配的基本单位 CPU 调度的基本单位
内存空间 拥有独立的地址空间 共享进程的地址空间（代码段、数据段、堆）
切换开销 大（需要切换页表、刷新 TLB） 小（只需保存寄存器、程序计数器）
通信方式 IPC（管道、共享内存、Socket） 直接读写共享内存（需同步机制）
创建开销 大（分配独立内存空间） 小（仅需分配栈和少量内核结构）

关键认知：线程是轻量级的进程，但共享内存带来了并发安全问题。Java 中线程的创建和销毁成本依然很高，生产环境绝对禁止直接 new Thread()，必须使用线程池。

2、并发 (Concurrency) vs 并行 (Parallelism)

并发关注”任务如何组织”，并行关注”任务如何执行”。高并发系统通常既需要并发设计（异步、解耦），也需要并行执行（多核利用）。

3、Java 线程的生命周期

状态说明：

  • NEW：线程刚创建，未调用 start()。
  • RUNNABLE：包含”就绪”和”运行”两种状态，等待 CPU 调度或正在执行。
  • BLOCKED：等待监视器锁（synchronized）进入阻塞状态。
  • WAITING：无限期等待（Object.wait()、Thread.join()、LockSupport.park()）。
  • TIMED_WAITING：限期等待（Thread.sleep()、Object.wait(long)、join(long)）。
  • TERMINATED：线程执行完毕或异常退出。

二、Java 内存模型 (JMM)：并发编程的基石

JMM 是 Java 并发编程的根基。如果不理解 JMM，你看再多源码也是”知其然而不知其所以然”。

1、为什么需要 JMM？

现代 CPU 为了提升性能，引入了三层优化：

  • CPU 寄存器：速度最快，容量最小。
  • CPU 缓存：每个核心有自己的 L1/L2 缓存，共享 L3 缓存。
  • 主内存：所有线程共享，但访问最慢。

问题来了：每个线程的工作内存（寄存器 + 缓存）是独立的，线程 A 修改了变量 x，线程 B 可能看不到最新值！这就是可见性问题。

JMM 的核心目标就是：定义线程和主内存之间的抽象关系，确保多线程环境下程序的正确性。

2、JMM 的三大特性

特性 含义 问题表现 解决方案
原子性 一个操作不可中断 i++ 不是原子操作（读取→修改→写入） synchronized、AtomicInteger、Lock
可见性 一个线程修改共享变量，其他线程立即可见 线程 A 修改了 flag，线程 B 看不到 volatile、synchronized、final
有序性 程序的执行顺序按照代码的先后顺序执行 指令重排导致执行顺序与代码不一致 volatile、synchronized、happens-before

3、volatile：最轻量的可见性保证

volatile 是 Java 中最轻量的同步机制，它能保证可见性和有序性（禁止指令重排），但不能保证原子性。

volatile 的内存语义：

  1. 写操作：线程修改 volatile 变量后，立即刷新到主内存。
  2. 读操作：线程读取 volatile 变量时，直接从主内存读取（使工作内存失效）。

volatile 的底层实现：

  • JVM 层面：在 volatile 读写操作前后插入内存屏障（Memory Barrier / Fence）。
  • 硬件层面：x86 架构通过 lock 前缀指令实现缓存一致性（MESI 协议）。

内存屏障类型：

volatile 的适用场景：

  • 状态标志位（如 boolean running = true 控制循环退出）。
  • 单例模式的 双重检查锁定（DCL）。
  • 不适用于计数器（i++ 非原子）等需要原子性的场景。

4、happens-before：JMM 的”法律条文”

happens-before 是 JMM 的核心规则，定义了操作之间的可见性约束。

规则 说明
程序次序规则 同一个线程中，前面的操作 happens-before 后面的操作
监视器锁规则 synchronized 解锁 happens-before 后续对同一把锁的加锁
volatile 规则 volatile 写 happens-before 后续对该变量的读
线程启动规则 Thread.start() happens-before 线程内的所有操作
线程终止规则 线程内的所有操作 happens-before 检测线程终止（Thread.join()）
中断规则 interrupt() happens-before 检测到中断事件
对象终结规则 构造函数执行 happens-before finalize()
传递性 A happens-before B，B happens-before C → A happens-before C

关键理解：happens-before 不是”时间上的先后顺序”，而是**”可见性上的保证”**。只要 A happens-before B，那么 A 的结果对 B 就是可见的。

5、CAS：无锁并发的基础

CAS（Compare-And-Swap，比较并交换）是 JUC 原子类的底层实现，也是乐观锁的核心。

CAS 的三大问题：

问题 说明 解决方案
ABA 问题 变量从 A → B → A，CAS 检测不到中间变化 AtomicStampedReference（加版本号）
自旋开销 竞争严重时，CAS 反复自旋，CPU 空转 自适应自旋、锁升级
只能保证单个变量 无法同时 CAS 多个共享变量 synchronized、Lock、Atomic 引用

三、synchronized：从”重量级锁”到”锁优化”

synchronized 是 Java 最基础的同步机制，JDK 6 之前它被称为”重量级锁”（每次加锁都涉及内核态切换）。但 JDK 6 引入锁升级机制后，synchronized 的性能已经媲美 ReentrantLock。

2、Monitor（监视器）原理

每个 Java 对象头中都包含一个 Monitor（监视器）结构，这是 synchronized 的底层实现：

lock 字段的取值：

3、锁升级的四阶段

偏向锁 (Biased Locking)：

  • 场景：只有一个线程访问同步块。
  • 原理：第一次获取锁时，在对象头中记录线程 ID。后续同一线程进入，只需检查线程 ID，无需 CAS。
  • 撤销：当第二个线程尝试获取锁时，撤销偏向锁，升级为轻量级锁。

轻量级锁 (Lightweight Locking)：

  • 场景：多个线程交替访问（无激烈竞争）。
  • 原理：线程在栈帧中创建 Lock Record，通过 CAS 将对象头替换为指向 Lock Record 的指针。获取失败则自旋（默认 10 次）。
  • 升级：自旋超过阈值或竞争激烈，升级为重量级锁。

重量级锁 (Heavyweight Locking)：

  • 场景：多线程竞争激烈。
  • 原理：通过操作系统 Mutex（互斥量）实现，线程阻塞和唤醒需要内核态切换，开销最大。

锁升级是单向的：一旦升级为重量级锁，不会再降级。但偏向锁撤销后可以重新偏向（JDK 15+ 默认关闭偏向锁，因为收益不如预期）。

4、synchronized 与 volatile 的区别

维度 synchronized volatile
原子性 保证 不保证（i++ 仍需原子类）
可见性 保证（解锁前刷新主内存） 保证（读写直接操作主内存）
有序性 保证（单线程内有序） 保证（禁止指令重排）
阻塞 会阻塞线程 不会阻塞线程
适用场景 复合操作（++、检查-然后-执行） 状态标志、单例 DCL

四、JUC 核心：AQS 与锁框架

JUC（java.util.concurrent）是 Java 并发编程的瑞士军刀。理解 JUC 的核心，就是理解 AQS（AbstractQueuedSynchronizer）。

1、AQS：并发同步的”骨架”

AQS 是一个抽象队列同步器，它定义了一套模板方法模式，让子类只需实现少量方法即可获得完整的同步功能。

AQS 的核心结构：

state 变量：

  • ReentrantLock：state = 0 表示未锁定，state > 0 表示重入次数。
  • CountDownLatch：state = 倒计数值，减到 0 时唤醒等待线程。
  • Semaphore：state = 可用许可证数量。

2、ReentrantLock：比 synchronized 更灵活的锁

ReentrantLock 的独家优势（synchronized 不具备）：

特性 ReentrantLock synchronized
可中断获取 lockInterruptibly() 不支持
超时获取 tryLock(long, TimeUnit) 不支持
非阻塞获取 tryLock() 不支持
公平/非公平 可配置（默认非公平） 只有非公平
条件变量 多个 Condition 只有一个 wait/notify
锁状态查询 isLocked()、getHoldCount() 不支持

公平锁 vs 非公平锁：

  • 非公平锁（默认）：允许”插队”。线程释放锁后，新到达的线程可能比队列中等待的线程先获取锁。吞吐量更高，但可能导致线程饥饿。
  • 公平锁：严格按照 FIFO 顺序获取锁。吞吐量略低，但避免饥饿。

生产建议：除非明确需要公平性，否则使用默认非公平锁（性能更好）。

3、CountDownLatch：”发令枪”同步器

适用场景：

  • 主线程等待多个子线程完成（如启动服务时等待多个组件初始化）。
  • 并行计算后汇总结果。

注意事项：CountDownLatch 是一次性的，计数器到 0 后不能重置。

4、Semaphore：限流的”许可证”

适用场景：

  • 数据库连接池：限制同时使用的连接数。
  • API 限流：限制并发请求数（如限流 100 QPS）。
  • 资源访问控制：如同时只允许 N 个线程访问文件。

5、CyclicBarrier：”循环栅栏”

与 CountDownLatch 的区别：

特性 CountDownLatch CyclicBarrier
等待方向 子线程等待主线程 线程间互相等待
可重用 一次性 可循环使用
计数器 递减到 0 递增到指定值
典型场景 主线程等子线程 多线程分阶段协作

五、线程池：生产环境唯一正确的线程使用方式

1、为什么必须用线程池？

直接 new Thread() 的问题：

  1. 资源消耗：每个线程默认占用 1MB 栈空间（64位 JVM），创建 1000 个线程 = 1GB 内存。
  2. 切换开销：线程过多导致 CPU 频繁上下文切换，反而降低吞吐量。
  3. 难以管理：线程生命周期、异常处理、回收都交给 JVM，不可控。

线程池的优势：

  • 复用线程：减少创建和销毁开销。
  • 控制并发数：防止资源耗尽。
  • 便于管理：线程数、任务队列、拒绝策略都可控。

2、ThreadPoolExecutor 的七大核心参数

参数详解：

参数 说明 生产建议
corePoolSize 常驻线程数，即使空闲也保留 CPU 密集型：CPU 核心数；IO 密集型：CPU 核心数 * 2
maximumPoolSize 最大线程数，队列满后创建新线程 核心数 * 2 或根据压测确定
keepAliveTime 非核心线程空闲多久后销毁 默认 60 秒，可自定义
workQueue 任务等待队列 有界队列优先（避免 OOM）
threadFactory 创建线程的工厂 推荐自定义，设置线程名前缀（便于排查）
handler 线程池和队列都满时的拒绝策略 根据业务选择（见下）

4、四种拒绝策略

策略 行为 适用场景
AbortPolicy 抛出 RejectedExecutionException（默认） 需要快速失败，通知调用者

5、四种内置线程池（不推荐直接用）

为什么不推荐？

  • newFixedThreadPool 和 newSingleThreadExecutor 使用 LinkedBlockingQueue（无界队列），任务堆积会导致 OOM。
  • newCachedThreadPool 的 maximumPoolSize = Integer.MAX_VALUE，线程无限创建会导致 OOM。

正确做法：使用 ThreadPoolExecutor 手动创建，配合有界队列（如 ArrayBlockingQueue）。

6、生产级线程池配置模板

关键原则：

  • IO 密集型：线程数可以多一点（CPU 等待 IO 时不占用 CPU）。
  • CPU 密集型：线程数 ≈ CPU 核心数（过多线程反而增加切换开销）。
  • 必须使用有界队列，防止无限堆积导致 OOM。
  • 必须自定义线程名，方便排查问题（Arthas 的 thread 命令直接看到线程池）。

为什么不用 Collections.synchronizedMap？

  • synchronizedMap：全局锁，所有读写操作串行，并发度 = 1。
  • ConcurrentHashMap：桶级别锁，并发度 ≈ 桶数量（默认 16）。

2、CopyOnWriteArrayList：读多写少的利器

核心思想：写操作时复制一份新数组，修改后替换原数组。读操作无锁，直接读原数组。

适用场景：读多写少（读占比 > 90%），如配置列表、黑名单、事件监听器列表。

注意：写操作开销大（O(n) 复制），不适合写频繁的场景。

七、并发编程高频面试题

1、volatile 能保证原子性吗？为什么？

不能。volatile 只能保证可见性和有序性，不能保证原子性。

i++ 操作实际上是三步：

  1. 读取 i 的值（volatile 保证读到最新值）。
  2. 执行 i + 1（在 CPU 寄存器中）。
  3. 写回 i（volatile 保证立即刷新到主内存）。

第三步依赖第一步读到的值，如果中间有其他线程修改了 i，结果就会出错。需要用 AtomicInteger 或 synchronized 保证原子性。

2、单例模式为什么要用 volatile？

instance = new Singleton() 在字节码层面分为三步：

  1. 分配内存空间。
  2. 初始化对象（构造函数）。
  3. 将引用指向分配的内存地址。

步骤 2 和 3 可能重排！如果线程 A 执行到步骤 3（此时 instance != null 但对象未初始化），线程 B 进入第一个 if，直接返回未初始化的对象，导致 NPE。

volatile 通过禁止指令重排（插入 StoreStore + StoreLoad 屏障）确保步骤 2 在步骤 3 之前完成。

3、synchronized 和 ReentrantLock 的区别？

维度 synchronized ReentrantLock
实现层面 JVM 关键字（Monitor） JDK 类（AQS）
释放方式 自动释放（异常也释放） 手动 unlock()（必须在 finally 中）
可重入 支持 支持
公平性 非公平 公平/非公平可选
可中断 不支持 lockInterruptibly()
超时获取 不支持 tryLock(timeout)
条件变量 一个 wait/notify 多个 Condition
性能 JDK 6+ 优化后接近 略高一点点（可忽略）

选择建议：简单场景用 synchronized（自动释放、JVM 优化），复杂场景（需要中断、超时、多条件）用 ReentrantLock。

4、线程池的核心参数有哪些？如何配置？

见”线程池”章节。核心参数 7 个：corePoolSize、maximumPoolSize、keepAliveTime、unit、workQueue、threadFactory、handler。

配置原则：

  • CPU 密集型：core = CPU 核心数，max = 核心数 + 1。
  • IO 密集型：core = CPU 核心数 * 2，max = 核心数 * 4（根据压测调整）。
  • 队列必须用有界队列，拒绝策略推荐 CallerRunsPolicy。

6、CountDownLatch 和 CyclicBarrier 的区别？

见”JUC 核心”章节。核心区别：CountDownLatch 是单向递减（一次性），CyclicBarrier 是线程间互相等待（可循环）。

7、什么是 AQS？它的核心设计思想是什么？

AQS（AbstractQueuedSynchronizer）是 JUC 中锁和同步器的底层框架。核心设计：

  • state 变量：表示同步状态（ReentrantLock 的锁状态、CountDownLatch 的计数器）。
  • FIFO 队列：维护等待获取同步状态的线程队列（CLH 变体）。
  • 模板方法：子类实现 tryAcquire/tryRelease，AQS 处理队列管理和线程阻塞/唤醒。

八、并发编程最佳实践与常见陷阱

1、生产环境并发编程的”十条戒律”

  1. 绝对禁止直接 new Thread()，全部使用线程池。
  2. 线程池必须用有界队列，LinkedBlockingQueue 无界会导致 OOM。
  3. 锁的粒度要细，不要用 synchronized 包裹整个方法，只锁定必要代码块。
  4. 优先使用 JUC 工具类，不要自己用 wait/notify 造轮子（容易出错）。
  5. ThreadLocal 用完后必须 remove()，线程池场景下尤其重要。
  6. 并发集合选择要谨慎：读多写少用 CopyOnWriteArrayList，高并发 Map 用 ConcurrentHashMap。
  7. 避免嵌套锁，防止死锁（锁顺序要一致）。
  8. 原子类优于 synchronized，简单的计数器用 AtomicLong 而不是锁。
  9. 线程名要自定义，出问题后用 Arthas 的 thread 命令能快速定位。
  10. 并发代码要单测覆盖，用 CountDownLatch 配合压测工具模拟并发场景。
