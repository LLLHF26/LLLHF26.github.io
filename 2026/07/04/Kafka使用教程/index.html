一、Kafka 核心基础概念

1. 核心组件对照表

名词 详细解释
Broker Kafka独立服务节点，一个容器=1个Broker；多Broker组成集群，分布式存储消息
Topic 消息逻辑分类主题，如order_topic专门存储订单消息
Partition 分区 Topic物理分片，一个Topic拆分多个分区分散在不同Broker；同一分区只能分配给同组一个消费者，保证分区内消息有序
Replica 副本 分区数据备份，集群必备；Leader副本负责读写，Follower同步数据；Leader宕机自动从可用副本选举新Leader
Offset 分区内消息自增唯一编号，作为消费游标，记录消费者读到哪条消息
ConsumerGroup 消费组 一组消费者实例，两条核心规则：
1. 同组：消息分摊，一条消息仅一个实例接收，用于业务并发扩容
2. 不同组：全量接收所有消息，用于多下游业务（库存/短信/数据统计）
Coordinator 协调者 每个消费组固定分配一台Broker，负责管理消费者加入、分区分配、Offset持久化

2. acks 可靠性参数说明

  • acks=0：不等待Broker应答，吞吐量最高，消息丢失风险极大
  • acks=1：仅等待Leader副本写入成功，Leader宕机、副本未同步完成会丢失消息
  • acks=-1(all)：等待ISR内所有副本同步完成才返回成功，生产环境推荐，集群副本数≥2才可生效

3. 核心铁则

  1. 单分区消息严格有序，多分区无法保证全局有序；
  2. 同消费组消费者数量 ≤ 分区总数，多余消费者闲置，收不到消息；
  3. 副本ISR集合：正在正常同步的副本列表，只有ISR内副本可参与Leader选举。

5. 双监听配置原理

  1. INTERNAL 监听：使用容器 hostname（kafka1/kafka2/kafka3），Docker 内部网络可互相解析，专门用于 Broker 之间副本同步、集群通信；
  2. EXTERNAL 监听：绑定宿主机127.0.0.1端口，供 Windows 本机 IDEA 中的 SpringBoot 项目连接；
  3. 旧配置仅使用127.0.0.1会导致跨容器副本同步失败，报Connection to 127.0.0.1:909X failed。

四、四大核心集群验证实验

实验 1：同消费组消息分摊（并发扩容）

  1. 复制项目，修改server.port=8081，库存消费组保持order-demo-group不变；
  2. 同时启动 8080、8081 两个实例；
  3. 连续发送多条订单消息；
  4. 现象：Topic 共 3 个分区，两个消费者自动分摊分区，一条消息只会被其中一个实例接收，用于业务水平扩容。

实验 2：Broker 宕机自动切换 Leader（集群高可用）

  1. 执行kafka-topics –describe查看目标分区当前 Leader Broker；
  2. 停止对应容器：docker stop kafka-broker3；
  3. 再次发送消息，控制台打印的 Leader 纪元会自动变更，集群选举新 Leader，消息正常收发无报错；
  4. 重启宕机 Broker：docker start kafka-broker3，节点自动同步缺失副本，重新加入 ISR 列表。

实验 3：acks 参数可靠性对比

  1. acks=1：仅写入 Leader 即返回，若写入后 Leader 立刻宕机，未同步副本的消息永久丢失；
  2. acks=-1：等待 ISR 全部副本同步完成才返回，集群单 Broker 宕机不会丢失消息，生产环境标准配置。

实验 4：auto-offset-reset 两种重置规则

  • latest：全新消费组，只读取项目启动后新产生的消息；
  • earliest：全新消费组，读取分区内全部历史消息；
  • Offset 持久化在 Broker，重启项目不会重复消费已提交 Offset 的消息。
