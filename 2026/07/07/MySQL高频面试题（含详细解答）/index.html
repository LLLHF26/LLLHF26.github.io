MySQL高频面试题

MySQL高频面试题

本套面试题涵盖MySQL基础、存储引擎、索引原理、事务与锁、SQL优化、主从复制、高频实战问题，适配初中级开发面试，答案详细通俗、贴合企业真实面试考点。

一、MySQL基础概念

1、MySQL常用存储引擎有哪些？InnoDB和MyISAM的核心区别？

解答：

MySQL主流存储引擎：InnoDB、MyISAM、Memory、Archive，其中InnoDB是MySQL5.5及以后默认存储引擎。

InnoDB与MyISAM核心区别：

  1. 事务支持：InnoDB支持事务、ACID特性；MyISAM不支持事务。

  2. 锁机制：InnoDB支持行级锁、表级锁、意向锁；MyISAM仅支持表级锁，并发性能差。

  3. 外键：InnoDB支持外键约束；MyISAM不支持。

  4. 崩溃恢复：InnoDB有redo/undo日志，支持故障自动恢复，数据安全；MyISAM无日志，宕机易数据损坏。

  5. 全文索引：MyISAM原生支持全文索引；InnoDB在MySQL5.6后支持全文索引。

  6. 性能场景：MyISAM查询速度快，适合读多写少、无事务场景；InnoDB适合读写均衡、需要事务、高并发场景。

2、什么是ACID？事务的四大特性分别是什么？

解答：

ACID是关系型数据库事务的四大核心特性，保障事务安全可靠：

  1. 原子性（Atomicity）：事务是最小执行单元，要么全部执行成功，要么全部回滚失败，不可部分执行。依托undo日志实现。

  2. 一致性（Consistency）：事务执行前后，数据库数据的完整性、约束性一致，不会出现非法数据（如转账总额不变、主键唯一等）。

  3. 隔离性（Isolation）：多个并发事务之间相互隔离，互不干扰，每个事务感知不到其他事务的执行状态。依托锁和MVCC实现。

  4. 持久性（Durability）：事务提交成功后，数据会永久写入磁盘，即使数据库宕机、服务器重启，数据也不会丢失。依托redo日志实现。

3、MySQL事务隔离级别有哪些？默认级别是什么？

解答：

MySQL提供4种事务隔离级别，级别越高，隔离性越强，并发性能越低，默认级别为可重复读（RR）。

  1. 读未提交（Read Uncommitted）：最低级别，能读取其他事务未提交的修改。会产生脏读、不可重复读、幻读问题，几乎不使用。

  2. 读已提交（Read Committed，RC）：只能读取其他事务已提交的数据。解决脏读，存在不可重复读、幻读，Oracle默认级别。

  3. 可重复读（Repeatable Read，RR）：事务内多次读取同一数据，结果一致。解决脏读、不可重复读，存在幻读，MySQL InnoDB默认级别。

  4. 串行化（Serializable）：最高级别，事务串行执行，完全隔离。解决所有并发问题，性能极低，极少使用。

核心问题解释：

  • 脏读：一个事务读取到另一个事务未提交的脏数据

  • 不可重复读：同一事务内，两次读取同一数据，结果不一致（其他事务更新并提交）

  • 幻读：同一事务内，两次范围查询，结果行数不一致（其他事务新增/删除数据并提交）

二、索引核心考点（高频重点）

1、MySQL索引的分类？B+树索引的优势？

解答：

索引分类：

  1. 按结构：B+树索引、哈希索引、全文索引

  2. 按约束：主键索引（唯一+非空）、唯一索引（唯一可空）、普通索引、联合索引

  3. 按存储：聚簇索引、二级索引（辅助索引）

B+树索引核心优势（对比B树、哈希）：

  1. 所有数据都在叶子节点，非叶子节点仅存索引键，内存占用小，磁盘IO少，查询效率高

  2. 叶子节点有序双向链表，支持范围查询、排序、分组，哈希索引仅支持等值查询

  3. 树层级低，查询稳定，每次查询磁盘IO次数固定

InnoDB聚簇索引规则：

  • 有主键：主键作为聚簇索引，叶子节点存储整行数据

  • 无主键：选择唯一非空字段作为聚簇索引

  • 无唯一字段：MySQL自动生成6字节隐藏rowid作为聚簇索引

2、什么是联合索引最左前缀原则？失效场景有哪些？

解答：

最左前缀原则：联合索引（a,b,c）遵循从左到右依次匹配，查询条件必须匹配最左侧的索引列，才能命中索引。

有效命中场景：a、a+b、a+b+c、a+c（仅a生效）

索引失效核心场景（高频）：

  1. 不满足最左前缀：查询条件无最左列，如查询b、c、b+c

  2. 索引列参与运算、函数、类型转换：where age+1=18、where DATE(create_time)='2026-01-01'

  3. 使用 !=、<>、not in、not like 否定查询

  4. like 以%开头：where name like '%张三'（右模糊可命中）

  5. 字符串字段未加引号，发生隐式类型转换

  6. or 前后字段有未索引字段（or两边需全部命中索引才生效）

  7. MySQL优化器判断全表扫描比索引查询更快，主动放弃索引（数据量少、重复度高）

3、覆盖索引、回表查询、索引下推是什么？

解答：

  1. 回表查询：二级索引叶子节点仅存储索引列+主键，查询字段不在索引中时，需要通过主键回到聚簇索引查询整行数据，产生二次IO，即为回表。

  2. 覆盖索引：查询的所有字段都包含在索引中，无需回表，直接通过二级索引获取全部数据，性能极高。
    示例：索引(a,b)，查询select a,b from table where a=1 即为覆盖索引。

  3. 索引下推（ICP）：MySQL5.6优化，在索引遍历阶段，直接过滤索引列的查询条件，减少回表次数，避免无效回表，提升查询效率。

4、为什么不建议使用过多索引？主键索引为什么建议自增？

解答：

索引过多的弊端：

  1. 索引占用磁盘空间，增加存储成本

  2. 增删改数据时，需要同步更新所有索引，降低写入性能

  3. 优化器选择索引耗时增加，可能导致索引选择错误

主键建议自增原因：

  1. 自增主键有序，B+树索引插入数据时，直接追加到叶子节点末尾，无页分裂、页合并，插入性能极高

  2. 无序主键（如UUID）会导致随机插入，频繁触发页分裂，索引结构碎片化，查询和写入性能大幅下降

  3. 自增主键占用空间小（int/bigint），二级索引存储主键，可减少索引体积

三、锁机制与MVCC（难点高频）

1、MySQL InnoDB有哪些锁？行锁和表锁的区别？

解答：

InnoDB锁分类：

  1. 按粒度：表级锁、行级锁、页级锁

  2. 按类型：共享锁（S锁、读锁）、排他锁（X锁、写锁）

  3. 辅助锁：意向共享锁、意向排他锁

行锁 vs 表锁：

  1. 表锁：锁定整张表，开销小、加锁快、无死锁，并发性能极差，MyISAM默认锁

  2. 行锁：锁定单行数据，开销大、加锁慢、可能死锁，并发性能极高，InnoDB默认锁

重点：InnoDB行锁生效前提是查询命中索引，未命中索引会退化为表锁！

2、什么是MVCC？实现原理是什么？如何解决幻读？

解答：

MVCC（多版本并发控制）：InnoDB实现无锁读的核心机制，通过数据版本快照，让读写互不阻塞，大幅提升并发性能，仅适用于RC、RR隔离级别。

核心三大隐藏字段：

  1. DB_TRX_ID：最后修改该行的事务ID

  2. DB_ROLL_PTR：回滚指针，指向undo日志的历史版本数据

  3. DB_ROW_ID：隐藏主键（无主键时生效）

实现原理：依托undo日志（历史数据版本）+ read view（读视图）

  1. 事务开启时生成read view，记录当前活跃事务ID

  2. 读取数据时，对比数据版本的事务ID，匹配规则读取当前可见版本

  3. 数据更新时，先拷贝旧数据到undo日志，生成新数据版本

RR级别解决幻读原理：

MySQL RR隔离级别通过**MVCC快照读 + 间隙锁+临键锁（当前读）**解决幻读：

  • 快照读（普通select）：依赖MVCC读取事务启动时的快照，看不到后续新增数据，无幻读

  • 当前读（select ... for update、update、delete）：通过间隙锁锁定区间，禁止其他事务新增数据，彻底杜绝幻读

3、什么是死锁？死锁产生的条件及解决方案？

解答：

死锁定义：两个或多个事务互相持有对方需要的锁，且互不释放，导致事务永久阻塞。

死锁四大必要条件：

  1. 互斥条件：锁同一时间仅一个事务持有

  2. 请求与保持：事务持有锁的同时，请求新的锁

  3. 不可剥夺：锁只能主动释放，不能被强制抢占

  4. 循环等待：多个事务形成锁等待环路

解决方案：

  1. 打破循环等待：统一SQL锁的获取顺序

  2. 缩短事务时长：避免长事务，尽早提交/回滚

  3. 设置超时时间：innodb_lock_wait_timeout，超时自动释放锁

  4. 开启死锁检测：InnoDB默认开启，自动回滚代价最小的事务

四、SQL优化实战（面试核心）

1、SQL慢查询优化的通用步骤？

解答：

  1. 定位慢SQL：开启慢查询日志，设置阈值（默认10秒），抓取耗时SQL

  2. 执行计划分析：使用explain查看SQL执行计划，重点关注type、key、rows、Extra字段

  3. 索引优化：补齐缺失索引、优化联合索引顺序、规避索引失效场景

  4. SQL语句优化：避免select *、减少子查询、优化join查询、避免大事务

  5. 架构优化：大表分库分表、读写分离、缓存兜底（Redis）

  6. 参数调优：调整缓冲池、连接数、超时时间等数据库参数

2、explain执行计划关键字段详解？

**解答：**核心字段（面试必考）：

  1. type：查询类型，性能从优到劣：system > const > eq_ref > ref > range > index > all
    核心要求：业务SQL尽量达到ref/range，杜绝all（全表扫描）

  2. key：实际命中使用的索引，为null表示未命中索引

  3. rows：预估扫描行数，数值越小性能越好

  4. Extra：额外信息
    Using filesort：文件排序，未命中索引排序，需要优化
    Using temporary：创建临时表，常见于分组、去重，性能差
    Using index：覆盖索引，无回表，性能最优

3、大表优化方案（千万级数据）？

解答：

  1. 索引优化：精简索引，保留高频查询索引，避免索引冗余；使用覆盖索引减少回表

  2. SQL优化：禁止select *、避免深分页、禁止大事务、拆分复杂SQL

  3. 分区表：按时间、ID分区，缩小查询扫描范围

  4. 分库分表：水平分表（拆分数据量）、垂直分表（拆分字段）、垂直分库（拆分业务）

  5. 读写分离：主库写、从库读，分摊数据库压力

  6. 冷热数据分离：历史冷数据归档至备份表，仅保留热数据在主表

五、主从复制与高可用

1、MySQL主从复制原理？复制延迟原因及解决方案？

解答：

主从复制核心流程：

  1. 主库（Master）数据写入、更新、删除时，记录操作日志到binlog二进制日志

  2. 从库（Slave）IO线程连接主库，拉取binlog日志，写入本地relay log中继日志

  3. 从库SQL线程读取relay log，重放日志操作，同步主库数据

主从延迟原因：

  1. 主库写入并发高，binlog生成速度大于从库重放速度

  2. 从库单线程重放日志（MySQL5.7前单线程），无法并行处理

  3. 大事务、大批量写入导致日志堆积

  4. 从库硬件性能低于主库

解决方案：

  1. 开启MySQL并行复制（5.7+支持）

  2. 拆分大事务，避免批量一次性写入

  3. 优化从库硬件、提升IO性能

  4. 业务层规避：核心读走主库、延迟数据做补偿逻辑

2、MySQL读写分离的实现方式？有什么坑？

解答：

实现方式：

  1. 代码层实现：业务代码手动区分读写，写主库、读从库

  2. 中间件实现：MyCat、Sharding-JDBC、ProxySQL自动路由读写请求

核心坑点：

  1. 主从延迟数据不一致：写入后立即查询，从库未同步完成，查询不到最新数据

  2. 从库故障导致读请求失败，需要做好故障降级

  3. 多从库数据同步差异，导致不同从库查询结果不一致

六、高频冷门实战面试题

1、limit深分页为什么慢？如何优化？

解答：

慢的原因：limit 100000,10 会先扫描前100010条数据，舍弃前100000条，仅返回后10条，扫描数据量极大，IO开销高。

优化方案：

  1. 主键分页优化（推荐）：利用主键有序特性，where id > 100000 limit 10，精准定位起始位置，无需扫描前置数据

  2. 覆盖索引分页：先通过索引查询主键，再关联查询整行数据

  3. 业务层限制：禁止前端深分页，限制最大页码

  4. 大数据量使用游标分页

2、redo日志和undo日志的区别和作用？

解答：

  1. redo log（重做日志）：
    作用：保证事务持久性，故障恢复时重做已提交事务，恢复数据
    时机：事务提交时写入，记录物理数据修改

  2. undo log（回滚日志）：
    作用：保证原子性+实现MVCC，事务回滚时撤销修改，同时存储历史数据版本
    时机：数据修改前写入，记录逻辑反向操作

3、MySQL如何避免幻读？

解答：

  1. 快照读（普通select）：依赖MVCC多版本控制，读取事务启动时的数据快照，规避幻读

  2. 当前读（update/delete/for update）：InnoDB RR级别通过间隙锁+临键锁锁定查询区间，禁止其他事务新增数据，彻底解决幻读

  3. 业务层面：串行化执行、悲观锁兜底
