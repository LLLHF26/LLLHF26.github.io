Redis 高频面试题

Redis 高频面试题及详细解答（完整版）

一、Redis 基础认知面试题

1. 什么是Redis？它的核心特性有哪些？

详细解答：Redis 是一款开源的、基于内存的、高性能的键值对（Key-Value）NoSQL 数据库，使用 ANSI C 语言编写，支持多种数据结构，常被用于缓存、限流、分布式锁、消息队列等场景。

核心特性：

  • 内存高效：核心数据存储在内存，读写速度极快，QPS 可达 10W+；支持磁盘持久化，防止数据断电丢失。

  • 多数据结构支持：不止简单的字符串，支持 List、Set、ZSet、Hash、Stream 等丰富数据结构，适配各类业务场景。

  • 单线程模型：核心读写命令单线程执行，避免线程竞争、锁开销，保证命令原子性。

  • 高可用、可扩展：支持主从复制、哨兵模式、Cluster 集群，实现故障自动转移、海量数据分片存储。

  • 功能丰富：支持过期删除、事务、Lua 脚本、发布订阅、流水线、分布式锁等高级功能。

  • 跨客户端兼容：支持 Java、Python、Go、PHP 等几乎所有主流开发语言。

2. Redis 为什么速度这么快？

详细解答：Redis 高性能是架构、IO、编码、模型多维度共同作用的结果，核心原因如下：

  1. 纯内存操作：所有数据读写都在内存中完成，避免磁盘 IO 的慢速开销，内存读写速度是磁盘的上千倍。

  2. 单线程无锁竞争：核心命令处理单线程串行执行，无需线程切换、锁竞争、死锁排查，执行效率极高。

  3. IO 多路复用模型：采用 epoll（Linux）、kqueue（Mac）多路复用技术，单线程可监听海量客户端连接，高效处理大量并发请求。

  4. 高效数据结构与编码：底层对每种数据结构做了深度优化，例如整数压缩编码、哈希压缩列表，减少内存占用和计算开销。

  5. 精简的协议设计：Redis 自定义 RESP 通信协议，文本格式简单、解析高效，无多余冗余数据。

  6. 流水线批量处理：支持 Pipeline 批量提交命令，减少网络 IO 往返次数，大幅提升批量操作性能。

补充：Redis 单线程仅指命令执行核心线程，持久化、集群心跳、文件读写等操作会使用多线程，不会阻塞核心业务。

3. Redis 支持哪些数据类型？各自的使用场景？

详细解答：Redis 常用 8 种数据类型，其中 5 种基础类型、3 种特殊类型，核心场景如下：

基础数据类型

  • String（字符串）：最基础类型，支持字符串、数字、二进制数据，最大 512MB。
    场景：缓存用户信息、验证码、token、计数器、限流、分布式ID。

  • Hash（哈希）：键值对集合，类似 Java 的 HashMap，适合存储结构化数据。
    场景：存储用户详情、商品信息、订单数据，相比 String 序列化更节省内存、支持局部更新。

  • List（列表）：有序可重复字符串链表，两头可读写。
    场景：消息队列、任务队列、用户点赞列表、最新资讯滚动列表。

  • Set（集合）：无序、不可重复字符串集合，支持交集、并集、差集。
    场景：用户关注列表、粉丝列表、共同好友、去重统计、抽奖活动。

  • ZSet（有序集合）：不可重复、按分值排序的集合。
    场景：排行榜、延时队列、权重排序、积分排名。

特殊数据类型

  • Stream（流）：Redis5.0+ 新增，可持久化消息队列，支持消息确认、消费者组。
    场景：可靠消息队列、日志收集、实时数据流处理。

  • Geo（地理位置）：存储经纬度坐标，支持距离计算、范围查询。
    场景：附近的人、附近门店、网约车位置匹配。

  • Bitmap/HyperLogLog：Bitmap 用于位运算统计，HyperLogLog 用于海量数据基数统计。
    场景：Bitmap 记录用户签到、活跃状态；HyperLogLog 统计页面UV、独立访客。

二、Redis 核心原理面试题

1. Redis 单线程模型详解，为什么不用多线程？

详细解答：

模型原理

Redis 核心的命令读取、解析、执行、返回结果全程由单个主线程完成，配合 IO 多路复用机制，实现单线程高并发处理。而 RDB/AOF 持久化、后台删除、集群同步等耗时操作，由独立子线程/后台线程执行，不阻塞主线程。

为什么不用多线程？

  1. 避免锁竞争开销：多线程操作共享内存数据，需要加锁保证线程安全，会带来锁等待、线程切换、死锁等问题，反而降低性能。

  2. CPU 不是瓶颈：Redis 绝大多数操作是内存读写，速度极快，瓶颈主要在网络 IO，单线程足以打满网络带宽，无需多线程提升CPU算力。

  3. 保证命令原子性：单线程串行执行命令，单个命令、Lua 脚本天然原子性，无需额外事务锁，逻辑简单可靠。

  4. 降低复杂度：单线程模型代码简洁，减少并发异常、数据不一致问题，稳定性更高。

补充：Redis6.0+ 引入多线程 IO，仅用于处理网络读写、协议解析，核心命令执行依旧单线程，兼顾并发性能和数据安全性。

2. Redis 过期键删除策略有哪些？优缺点对比？

详细解答：Redis 针对过期 Key 采用惰性删除 + 定期删除的混合策略，杜绝内存泄漏和CPU浪费。

（1）惰性删除

原理：Key 过期后不会立即删除，下次访问该Key时才检查过期并删除。

优点：极致节省CPU资源，无需后台扫描，只处理需要访问的过期键。

缺点：大量过期Key长期不被访问，会一直占用内存，导致内存泄漏。

（2）定期删除

原理：Redis 每隔一段时间（默认100ms），随机抽取部分过期Key扫描并删除，限制扫描时长和数量，避免阻塞主线程。

优点：弥补惰性删除的内存泄漏问题，定时释放过期内存。

缺点：无法删除所有过期Key，存在遗漏；扫描耗时过长会轻微阻塞主线程。

（3）兜底策略：内存淘汰机制

当内存达到 maxmemory 上限时，触发内存淘汰，强制清理数据，彻底解决过期Key残留问题。

3. Redis 内存淘汰策略有哪些？各自适用场景？

详细解答：Redis 共 8 种内存淘汰策略，默认 noeviction，分为 4 大类，核心适配场景如下：

策略名称 规则说明 适用场景
noeviction（默认） 内存满了不删除数据，直接报错拒绝写入 数据零丢失场景，不允许缓存淘汰
allkeys-lru 淘汰所有Key中最近最少使用的 通用缓存场景，优先保留热点数据（最常用）
volatile-lru 只淘汰带过期时间的Key中最少使用的 部分数据永久存储，仅缓存临时数据
allkeys-lfu 淘汰所有Key中使用频率最低的 长期热点缓存，规避LRU缓存抖动问题
volatile-lfu 只淘汰带过期时间、使用频率最低的 临时数据缓存，精准保留高频访问数据
allkeys-random 随机淘汰任意Key 数据访问无规律、对缓存命中率要求低
volatile-random 随机淘汰带过期时间的Key 仅需清理临时过期数据，无访问优先级
volatile-ttl 优先淘汰剩余过期时间最短的Key 需要快速释放即将过期的缓存数据

生产推荐：通用业务优先使用 allkeys-lru，热点稳定业务使用 allkeys-lfu。

三、Redis 持久化面试题

1. Redis 两种持久化机制 RDB 和 AOF 原理、优缺点、生产选择？

详细解答：Redis 持久化用于解决断电、进程退出导致内存数据丢失问题，提供 RDB、AOF 两种机制，可单独使用或混合开启。

1. RDB（快照持久化）

原理：定期将当前内存中的全量数据生成二进制快照文件（.rdb），保存到磁盘。触发方式分为手动（save、bgsave）和自动（配置规则定时触发）。

优点：

  • 文件体积小、压缩率高，适合冷备份、数据迁移；

  • 恢复速度极快，直接加载二进制文件即可；

  • 后台子线程执行，几乎不阻塞主线程。

缺点：

  • 存在数据丢失风险：两次快照之间的增量数据无法保存，故障会丢失最近一段时间数据；

  • 频繁快照会消耗磁盘IO、CPU资源。

2. AOF（日志持久化）

原理：记录每一条写操作命令到日志文件（.aof），重启服务时重放日志命令恢复数据，采用“先写内存、后写磁盘”机制。

优点：

  • 数据安全性极高，支持秒级甚至几乎零数据丢失；

  • 日志文件可追加、可重写，容错性高。

缺点：

  • 文件体积远大于RDB，长期运行日志臃肿；

  • 数据恢复速度慢，需要逐条重放命令；

  • 高频写入时磁盘IO开销较大。

3. 生产环境选择方案

  • 数据零丢失场景：开启 AOF + 每秒刷盘策略；

  • 高性能、允许少量数据丢失：单独开启 RDB；

  • 主流生产方案：RDB+AOF 混合开启，兼顾备份速度和数据安全性，Redis重启优先加载AOF文件恢复数据。

2. AOF 重写机制是什么？为什么需要重写？

详细解答：

重写原理

AOF 持续追加写命令会导致日志文件无限膨胀，Redis 会触发AOF重写：后台子线程根据当前内存数据，生成一套最简可恢复数据的命令日志，替换原有臃肿的AOF文件。重写过程不会删除旧日志，新日志生成后才完成替换，保证数据安全。

重写原因

  • 大量无效命令堆积：对同一个Key的多次修改、删除、覆盖操作，无需全部记录；

  • 缩减文件体积，减少磁盘占用，提升数据恢复速度；

  • 避免日志文件过大导致的IO性能下降。

触发方式：自动触发（文件大小超过阈值）、手动触发（bgrewriteaof 命令）。

四、Redis 缓存经典问题面试题（缓存三大问题）

1. 缓存穿透、缓存击穿、缓存雪崩区别、成因、解决方案？

（1）缓存穿透

定义：查询数据库和缓存都不存在的数据，请求直接穿透到数据库，大量恶意请求会压垮数据库。

成因：用户查询非法ID、空数据、恶意爬虫攻击。

解决方案：

  • 空值缓存：查询为空时，缓存空Key，设置短期过期时间；

  • 布隆过滤器：提前过滤不存在的Key，拦截非法请求；

  • 接口校验：参数合法性校验、限流、黑名单拦截。

（2）缓存击穿

定义：单个热点Key过期瞬间，大量并发请求同时打向数据库，瞬间压垮DB。

成因：热点缓存Key过期、瞬时并发量大。

解决方案：

  • 互斥锁（分布式锁）：同一时间仅一个请求查询DB、更新缓存；

  • 热点Key永不过期：手动取消过期时间，后台异步更新缓存；

  • 过期时间随机偏移：避免批量热点Key同时过期。

（3）缓存雪崩

定义：大量缓存Key同时过期 / Redis服务宕机，所有请求全部访问数据库，导致数据库雪崩宕机、服务瘫痪。

成因：批量Key过期时间一致、Redis集群故障、断电、网络波动。

解决方案：

  • 过期时间加随机值，打散过期时间，避免集中失效；

  • 搭建 Redis 高可用集群（主从+哨兵/Cluster），避免单点故障；

  • 服务层限流、降级、熔断，保护数据库；

  • 多级缓存架构（本地缓存+Redis分布式缓存），兜底容错。

2. 缓存和数据库双写一致性问题如何解决？

详细解答：缓存与数据库数据不一致是业务高频问题，核心是选择合适的更新策略，适配不同一致性需求。

1. 常规策略对比

  • 先更新数据库，再更新缓存：并发更新会导致脏缓存，极少使用；

  • 先更新缓存，再更新数据库：数据库更新失败会导致数据完全不一致，禁止使用；

  • 先更新数据库，再删除缓存（主流方案）：业务首选，一致性最优。

2. 主流方案：先更库、后删缓存

流程：业务更新数据库成功 → 删除对应缓存Key → 下次查询自动加载最新数据。

存在问题：极低概率出现「读请求先查缓存（旧数据），写请求删缓存后，读请求未更新缓存」的短暂不一致。

优化方案：

  • 延迟双删：更新完成后立即删缓存，延迟500ms再次删除，覆盖脏数据；

  • 异步删除：通过MQ异步删除缓存，提升性能、保证最终一致。

3. 强一致性方案

对金融、支付等强一致场景，使用分布式锁：读写请求竞争同一锁，串行执行，杜绝并发不一致问题。

4. 最终一致性兜底

定时任务轮询比对数据库和缓存数据，自动修复不一致数据。

五、Redis 高可用面试题

1. Redis 主从复制原理、完整流程？

详细解答：主从复制是 Redis 高可用的基础，实现主节点写、从节点读，数据实时同步，读写分离，提升并发能力。

完整复制流程

  1. 建立连接：从节点启动后，向主节点发送SYNC命令，建立网络连接。

  2. 全量同步：主节点收到SYNC后，执行bgsave生成RDB快照，同时缓存期间所有写命令；将RDB文件发送给从节点，从节点加载快照数据。

  3. 增量同步：快照同步完成后，主节点将缓存的增量写命令批量发送给从节点，从节点执行命令，保证数据一致。

  4. 持续同步：后续主节点所有写操作，实时同步到从节点，保持主从数据一致。

复制优化（Redis2.8+）

支持部分重同步：主从断连重连后，无需全量同步，通过复制偏移量和积压日志，仅同步断连期间的增量数据，大幅提升同步效率。

2. 哨兵模式（Sentinel）原理、作用、故障转移流程？

详细解答：哨兵是 Redis 官方高可用方案，基于主从复制实现自动故障检测、自动主从切换、监控告警，解决主节点单点故障问题。

哨兵核心作用

  • 监控：实时监控主、从节点在线状态；

  • 通知：节点故障时推送告警信息；

  • 自动故障转移：主节点宕机，自动选举新主节点，同步数据、更新集群配置。

故障转移完整流程

  1. 主观下线：单个哨兵检测到主节点超时无响应，标记为主观下线；

  2. 客观下线：多个哨兵（达到法定票数）确认主节点故障，标记为客观下线，触发故障转移；

  3. 选举哨兵领导者：所有哨兵投票选出一个领导者，负责执行切换操作；

  4. 选举新主节点：从所有从节点中，筛选数据完整、延迟最低的节点作为新主；

  5. 切换主从：将原主节点降级为从节点，挂载到新主节点下；

  6. 客户端重定向：哨兵推送新主节点地址，客户端自动切换连接。

3. Redis Cluster 集群原理、哈希槽、分片规则？

详细解答：Redis Cluster 是 Redis 分布式集群方案，支持数据分片、横向扩容、故障自愈，解决单节点内存上限、并发上限问题，生产海量数据首选。

核心原理

Cluster 采用哈希槽分片机制，整个集群共有 16384 个哈希槽（0-16383），每个主节点负责一部分哈希槽，所有Key通过哈希算法映射到对应槽位，实现数据分布式存储。

Key 分片规则

  1. 对 Key 进行 CRC16 哈希运算，得到哈希值；

  2. 哈希值对 16384 取模，得到对应的哈希槽编号；

  3. 根据槽位编号，路由到对应主节点读写数据。

集群特性

  • 最小集群规模：3主3从，保证高可用；

  • 支持横向扩容：新增节点可迁移哈希槽，无需停机；

  • 故障自愈：单个主节点故障，从节点自动升级为主节点；

  • 支持哈希Tag：通过{}包裹Key部分，实现同一槽位数据聚合，适配批量操作。

六、Redis 分布式锁面试题

1. Redis 分布式锁实现原理、优缺点？

详细解答：分布式锁用于解决多服务、多线程并发竞争资源问题，Redis 分布式锁是目前业务最常用的实现方式。

核心实现命令

SET key unique_value NX EX timeout

  • NX：仅Key不存在时设置成功，保证抢锁原子性；

  • EX：设置锁过期时间，防止死锁；

  • unique_value：唯一随机值，防止误删他人锁。

加锁、解锁流程

  1. 加锁：客户端执行SET命令，设置锁Key、唯一值、过期时间，执行成功即抢到锁；

  2. 执行业务逻辑：抢到锁的客户端执行核心业务；

  3. 解锁：通过Lua脚本校验当前锁的唯一值，匹配成功则删除Key，释放锁（保证解锁原子性）。

优缺点

优点：性能极高、实现简单、支持超时防死锁、适配分布式场景。

缺点：主从异步复制存在锁失效风险（主节点加锁成功，未同步到从节点就宕机，从节点升级后主锁失效）。

2. Redlock 红锁算法原理、适用场景？

详细解答：Redlock 是 Redis 官方提出的高可靠分布式锁算法，解决单节点分布式锁的数据不一致问题。

算法原理

部署奇数个独立Redis主节点（5个最佳），加锁时向所有节点依次申请锁，只要超过半数节点加锁成功，即视为抢锁成功；解锁时同时删除所有节点的锁Key。

核心特性

  • 无需主从复制、集群机制，节点相互独立；

  • 彻底规避主从同步延迟导致的锁失效问题；

  • 容错性高，少量节点故障不影响锁正常使用。

适用场景

金融、支付、订单等对锁安全性要求极高的场景；普通业务无需使用，性能开销大、部署复杂。

七、Redis 高频进阶面试题

1. Redis 事务原理，为什么不支持回滚？

详细解答：Redis 事务是一组命令的批量执行，核心保证批量命令串行、无插队执行。

事务流程

MULTI（开启事务）→ 入队所有命令 → EXEC（批量执行）/ DISCARD（放弃事务）。

为什么不支持事务回滚？

  • Redis 事务错误分为两种：语法错误（整体放弃执行）、运行时错误（执行失败不回滚）；

  • 设计者认为 Redis 事务错误均为编码问题，可在开发阶段规避，无需运行时回滚；

  • 放弃回滚机制，极大简化代码、提升执行性能，贴合Redis高性能定位。

补充：需要强原子性、事务回滚场景，可使用 Lua 脚本替代事务。

2. Redis Pipeline 和 Lua 脚本的区别和使用场景？

详细解答：两者均用于批量处理命令、减少网络IO开销，核心差异在原子性和灵活性。

Pipeline（流水线）

原理：批量打包多个命令，一次性发送给Redis，一次性返回所有结果，减少网络往返次数。

特点：无原子性，命令独立执行，部分失败不影响其他命令；性能高、开销小。

场景：海量简单命令批量读写、数据初始化、批量删除。

Lua 脚本

原理：将多条命令封装为一个Lua脚本，Redis一次性执行整个脚本，天然原子性，脚本内命令全部执行成功或全部失败。

特点：支持逻辑判断、循环、变量运算，原子性强，灵活性极高。

场景：分布式锁解锁、复杂批量操作、限流、事务级批量处理。

3. Redis 热点Key、大Key问题如何排查和解决？

详细解答：大Key、热点Key是Redis生产最常见的性能瓶颈，会导致CPU飙升、响应超时、集群卡顿。

1. 大Key（Value过大/元素过多）

危害：读写耗时变长、网络IO阻塞、内存分配卡顿、集群迁移超时。

排查：redis-cli --bigkeys、scan 遍历、监控key占用内存。

解决方案：

  • 拆分大Key：将大Hash/List拆分为多个小Key；

  • 精简数据：剔除无效字段、缩短存储内容；

  • 分批操作：避免一次性读写全量数据，分页获取。

2. 热点Key（超高并发访问单个Key）

危害：单节点QPS过载、CPU打满、集群负载不均衡。

排查：监控热点接口、redis监控平台、日志统计高频Key。

解决方案：

  • 本地缓存兜底：JVM本地缓存+Redis，减少Redis访问压力；

  • Key分片：热点Key拆分多个副本Key，分散并发流量；

  • 永久缓存：静态热点数据取消过期时间，避免集中缓存重建。
