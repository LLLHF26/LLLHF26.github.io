一条Docker误指令，清空所有容器！重配MySQL/Redis全过程复盘

一条Docker误指令的代价！

【踩坑日记】一条Docker误指令，清空所有容器！重配MySQL/Redis全过程复盘

记录时间：2026年07月05日

场景：WSL2 Ubuntu 环境部署 ELK 日志分析栈

事故核心：误用 Docker 清理指令，误删除全部业务容器，被迫重新部署 MySQL8.0、Redis6 服务

一、事故背景

今天计划在本地 WSL2 开发环境中，搭建 ELK（Elasticsearch+Logstash+Kibana） 日志收集分析平台，用于学习日志可视化、服务日志排查。

在此之前，我的 WSL Ubuntu 环境已经稳定运行了两个核心容器：

  • MySQL 8.0.43：挂载目录至 /mydata/mysql，数据持久化，用于本地项目数据库存储

  • Redis 6-alpine：挂载目录至 /mydata/redis，用作缓存服务

两个服务均完成目录挂载、密码配置、开机自启，原本可以直接使用，无需重复部署。本以为部署 ELK 是简单的镜像拉取、容器启动操作，没想到因为指令记忆模糊、操作粗心，直接引发翻车事故。

二、事故发生全过程

1. 误操作指令

部署 ELK 前，为了清理环境中停止的无用容器、释放资源，我想要执行「清理停止的容器」指令，但是手滑输错命令，错误执行了：

也有另一种失误可能：混淆了 prune 系列指令范围，最终结果就是：系统中所有已停止、甚至部分运行中容器被强制清除。

2. 事故结果

执行完指令回车的瞬间，终端快速刷新日志，我瞬间意识到不对劲，立刻执行docker ps -a 查看所有容器：

结果空空如也！MySQL、Redis 容器全部被删除，之前配置好的所有业务容器直接清空。

万幸中的万幸：我之前做了目录挂载持久化！

容器被删除，但 WSL 本地 /mydata/mysql、/mydata/redis 中的数据文件、配置文件完全保留，没有丢失任何业务数据，这是本次事故最大的止损点。

三、问题根源：指令认知误区

复盘本次翻车，核心是对 Docker 清理指令理解不透彻，特此记录区分，避免下次再犯：

错误指令 指令作用 正确使用场景
docker container prune -f 强制清理所有停止状态的容器，无二次确认 仅全新环境清理垃圾使用，绝对不能在有业务容器的环境执行
docker system prune 清理停止容器、无用镜像、网络、缓存 风险极高，生产/开发环境慎用

我的失误核心：想只清理垃圾容器，却忽略了该指令会清空所有停机容器，而我之前为了测试偶尔启停服务，部分容器处于停止状态，直接被批量清空。

四、补救修复：重新部署容器

因为数据和配置文件全部挂载在本地目录未丢失，无需重新配置 my.cnf、redis.conf，也无需恢复数据，只需要重新创建容器、挂载原有目录即可快速恢复服务。

3. 服务验证

重启完成后，执行 docker ps，MySQL、Redis 正常运行，Windows 通过 \\wsl.localhost\Ubuntu\mydata\ 访问挂载目录，数据、配置完全保留，服务恢复正常。

五、核心复盘与经验总结（重点）

本次虽然没有造成数据损失，但浪费了大量部署时间，暴露了操作不严谨、指令不熟练的问题，总结几条必须遵守的规范：

  1. 容器数据挂载是底线
    本次事故最大的保命操作就是 所有服务必做目录挂载持久化。容器是临时的，随时可能被删除、销毁，数据绝对不能存于容器内部，必须挂载到宿主机本地目录。

  2. 高危 prune 指令禁止盲敲prune 系列是 Docker 高危清理指令，开发环境非必要不使用，使用前必须先执行 docker ps -a 确认所有容器状态。

  3. 操作前备份/确认环境
    涉及容器删除、环境清理的操作，一律先确认当前运行的业务服务，杜绝无脑执行命令。

  4. 常用服务部署脚本留存
    本次快速恢复的原因是留存了部署命令，后续将把 MySQL、Redis、Nginx 等常用服务的 docker run 命令整理成脚本，翻车可一键恢复。

六、后续计划

解决容器翻车问题后，继续完成本次核心任务：ELK 日志栈部署搭建，同时后续会单独出一篇 ELK 部署踩坑笔记，记录完整的部署流程与问题解决方案。

谨记：技术踩坑不可怕，不复盘、不总结才是最大的问题！
