Kubernetes 实战详解：从核心架构到生产部署全指南

一、为什么需要容器编排？Docker 之后的下一站

上一篇踩坑日记复盘了 Docker 误指令清空容器的惨痛经历。但 Docker 解决的只是「单机单容器」的运行问题——到了生产环境，几十上百个微服务跑在几十台机器上，Docker 独木难支：

痛点 Docker 单机的局限 K8s 的解法
容器挂了没人管 手动 docker restart，半夜宕机全靠运维盯盘 自动重启 + 健康检查 + 自愈
扩缩容全靠人工 加机器、拉镜像、改端口，流程繁杂 kubectl scale 一行命令，HPA 自动伸缩
服务发现靠脑子记 IP 端口散落在笔记本里，改一下就全乱 Service + DNS 自动发现，零配置
滚动更新靠运气 先停旧容器再拉新镜像，中间有停机窗口 滚动更新 + 就绪探针，零停机切换
配置管理靠 SSH 每台机器手动改环境变量，改漏一台就翻车 ConfigMap / Secret 统一分发，热更新

一句话总结：Docker 把应用装进了集装箱，Kubernetes 则是整个集装箱码头的调度系统。

二、核心架构：控制平面 + 工作节点

K8s 集群由两类节点组成：

1、控制平面（Control Plane）——集群的大脑

组件 职责 一句话定位
kube-apiserver 集群唯一入口，所有操作都经它 REST API “前台接待”，所有请求的入口与校验
etcd 分布式 KV 存储，存集群所有状态数据 “档案室”，集群状态的唯一权威数据源
kube-scheduler 决定 Pod 跑在哪个 Node 上 “调度员”，根据资源/亲和性/污点给 Pod 分配座位
kube-controller-manager 运行各种控制器（Deployment/ReplicaSet/Node 等） “巡视员”，持续对比期望状态与实际状态，驱动收敛
cloud-controller-manager 与云厂商 API 交互（节点/路由/服务/卷） “外交官”，对接 AWS/GCP/阿里云等云平台

2、工作节点（Worker Node）——干活的手

组件 职责 一句话定位
kubelet 管理 Pod 生命周期，向 apiserver 上报节点状态 “车间主任”，接收调度指令、启停容器、健康上报
kube-proxy 维护 Service 的网络规则（iptables / IPVS） “网络工程师”，实现 Service 负载均衡与端口映射
容器运行时 实际跑容器（containerd / CRI-O，Docker 已弃用） “生产线”，真正执行容器启停

3、架构交互全景

核心设计哲学：声明式 + 控制循环

你告诉 K8s “我要 3 个 Pod”，控制器就会不断循环——发现只有 2 个就补 1 个，发现有 4 个就删 1 个，直到实际状态与期望状态一致。这就是 K8s 自愈能力的根源。

三、六大核心资源对象：设计哲学与实战用法

1、Pod —— 最小调度单元

Pod 不是「一个容器」——它是一组共享网络和存储的容器集合。同一个 Pod 内的容器：

  • 共享同一个 IP 地址和端口空间
  • 共享同一个 Volume
  • 可以通过 localhost 互相通信

restartPolicy Always（默认）/ OnFailure / Never

5、PV 与 PVC —— 存储抽象

PersistentVolume（PV）是集群级的存储资源，PersistentVolumeClaim（PVC）是 Pod 级的存储申请——类比「机房硬盘」和「租户申请单」：

accessMode 含义
ReadWriteOnce (RWO) 单 Node 读写
ReadOnlyMany (ROX) 多 Node 只读
ReadWriteMany (RWX) 多 Node 读写

StorageClass + 动态供给：PVC 不指定 PV 名，K8s 根据 StorageClass 自动创建 PV——这是生产主流做法。

五、健康检查：自愈的核心机制

K8s 提供三种探针（Probe），这是容器自愈的根基：

探针类型 作用 失败后果
livenessProbe 检查容器是否存活 失败 → 重启容器（Kill + Restart）
readinessProbe 检查容器是否就绪（能接流量） 失败 → 从 Service Endpoints 移除
startupProbe 检查慢启动应用是否完成初始化 失败 → Kill 容器（给慢应用更长的启动宽限期）

最佳实践：

  • livenessProbe 和 readinessProbe 要分开：liveness 只查「进程活着」，readiness 查「业务就绪」
  • 千万别把慢启动逻辑放在 livenessProbe 里——容器还没启动完就被 Kill 了，进入 CrashLoopBackOff 死循环。用 startupProbe 保护慢应用。
  • readinessProbe 失败不重启容器——只是暂时摘掉流量，等恢复自动加回。

十、2026 年 Kubernetes v1.36 新特性速览

Kubernetes v1.36（代号”Haru”，2026 年 4 月发布）带来了多项重要演进：

特性 状态 说明 对生产的影响
Pod 级资源管理器 Alpha kubelet 资源管理从按容器分配 → 支持 Pod 级 .spec.resources 整体规约 ML/GPU、低延迟场景可更精确管控整机资源
Pod 级原地垂直扩缩容 Beta 运行中 Pod 可动态调整资源配额，通常无需重启容器 HPA 垂直伸缩更平滑，减少因调整资源导致的 Pod 重启
Gateway API GA 替代 Ingress 的新一代流量入口标准，多角色 + 跨命名空间 + TCP/UDP Ingress-NGINX 已于 2026 年 3 月退休，必须迁移
DRA 动态资源分配 Beta 进阶 GPU/加速器等异构资源更灵活分配，支持优先级列表 + 可切分设备 GPU 共享、多型号回退调度成为标准能力
卷组快照 GA 可同时为多个 PVC 创建崩溃一致性快照 数据库集群备份、一致性恢复从 hack 变成标准 API
Memory QoS Alpha 分层内存保护，关键业务有内存保障、次要业务可被限流 混部场景（在线 + 离线跑同一集群）更安全
工作负载感知调度 Alpha PodGroup API 原子调度，支持拓扑感知 + 工作负载感知抢占 AI/ML 批任务不再”散落”调度，提升集群利用率
基于清单的准入控制 Alpha 准入 Webhook 和 CEL 策略可在集群引导时从文件加载 安全策略从集群启动时就生效，消除策略真空窗口

生产迁移提醒：如果你的集群仍在用 Ingress-NGINX，2026 年必须迁移到 Gateway API。Ingress-NGINX 已于 2026 年 3 月正式退休，不再收到安全更新和 bug 修复。

十一、高频面试题

1、Pod 和 Deployment 的关系是什么？

Pod 是最小调度单元，Deployment 是声明式副本控制器。Deployment 通过 ReplicaSet 管理 Pod 数量——你声明 replicas: 3，控制器确保始终有 3 个 Pod 运行。更新镜像时 Deployment 自动滚动更新，回滚有历史版本记录。

2、Service 的四种类型及适用场景？

  • ClusterIP：集群内部访问，默认类型，微服务间互调
  • NodePort：在每台 Node 开高位端口（30000-32767），临时外部暴露
  • LoadBalancer：云厂商自动创建外部 LB，生产级对外暴露
  • ExternalName：CNAME 到外部域名，引用集群外服务

3、livenessProbe 和 readinessProbe 有什么区别？

  • livenessProbe 失败 → 重启容器：确保进程活着
  • readinessProbe 失败 → 从 Service Endpoints 移除：确保业务就绪才接流量
  • 关键区别：readinessProbe 失败不重启容器，只是暂时摘掉流量。如果混用（把业务逻辑放在 liveness 里），会导致慢启动应用被反复 Kill。

4、如何实现零停机滚动更新？

Deployment 默认 RollingUpdate 策略 + readinessProbe 配合：

  1. 新 Pod 启动 → readinessProbe 检查通过 → 加入 Service Endpoints
  2. 旧 Pod 从 Endpoints 移除 → 优雅终止（SIGTERM + terminationGracePeriodSeconds）
  3. maxUnavailable: 0 确保更新过程中始终有足够 Pod 接流量

5、ConfigMap 和 Secret 的区别与安全注意事项？

  • ConfigMap 存普通配置，Secret 存敏感信息（密码/密钥）
  • Secret 的 data 字段是 base64 编码，不是加密——任何人都能 base64 -d 解码
  • 生产安全：RBAC 限制 Secret 访问 + etcd 开启 --encryption-provider-config 加密存储 + 使用 Vault / Sealed Secrets

6、HPA 扩缩容不生效怎么办？

检查三项：

  1. Pod 是否设置了 resources.requests（HPA 利用率 = 实际用量 / requests，没 requests 就没利用率）
  2. Metrics Server 是否安装（kubectl top pods 能返回数据才算安装成功）
  3. behavior.scaleDown.stabilizationWindowSeconds 是否设置过长（冷却期太长缩容慢）

7、Ingress 和 Gateway API 的区别？为什么 2026 要迁移？

维度 Ingress Gateway API
角色模型 单一管理员 多角色（Infra/Cluster/App）
扩展性 注解 hack 丰富字段 + 自定义
多命名空间路由 不支持 原生支持
TCP/UDP 不支持 原生支持
金丝雀权重 注解 hack 原生 weight 字段

Ingress-NGINX 于 2026 年 3 月退休，不再有安全更新。必须迁移到 Gateway API。

十二、生产避坑清单

遱坑 后果 防范
Pod 不设 resources.requests HPA 不生效、调度器无法判断 Node 容量 所有 Pod 必设 requests + limits
Secret 只用 base64 不加密 任何人可解码 etcd 加密存储 + RBAC + Vault
livenessProbe 和 readinessProbe 用同一个路径 慢启动应用被反复 Kill 分开：liveness 只查存活，readiness 查就绪
不设 terminationGracePeriodSeconds 强制 Kill 导致数据丢失 设 30-60s + 应用捕获 SIGTERM 优雅退出
所有 Pod 跑在同一 Node Node 故障全部挂 podAntiAffinity + 多副本分散
NodePort 端口冲突 多个 Service 争同一端口 生产用 LoadBalancer / Gateway API
etcd 没有备份 集群状态全丢 定期 etcdctl snapshot save + 异地备份
滚动更新不设 maxUnavailable: 0 更新中间有停机窗口 就绪探针 + maxUnavailable=0
忽略 ImagePullPolicy 本地旧镜像不是最新 生产用 Always，本地开发用 IfNotPresent
用 Docker 作为容器运行时 K8s v1.24 已弃用 Docker 使用 containerd 或 CRI-O
