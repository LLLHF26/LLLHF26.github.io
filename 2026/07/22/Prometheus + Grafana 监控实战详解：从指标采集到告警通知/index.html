Prometheus + Grafana 监控实战详解：从指标采集到告警通知

一、为什么后端工程师必须懂监控

线上系统出问题的那一刻，最怕的不是 Bug 本身，而是”两眼一抹黑”——不知道哪个服务慢了、哪台机器满了、哪个接口在报错。监控（Monitoring）就是后端工程师的”夜视仪”，它回答三个核心问题：

  1. 系统现在是否正常？（健康检查、存活探针）
  2. 问题出在哪里？（指标定位、链路追踪、日志关联）
  3. 如何提前发现？（阈值告警、趋势预测、异常检测）

现代可观测性（Observability）体系通常由 Metrics（指标）、Logs（日志）、Traces（链路追踪） 三大支柱组成，Prometheus + Grafana 正是 Metrics 领域的事实标准。CNCF 毕业的 Prometheus 已被 Kubernetes、Docker、etcd、CoreDNS 等云原生项目作为默认监控方案，Grafana 则是业界最流行的可视化平台。

能力 传统脚本/命令 Prometheus + Grafana
采集方式 手动登录服务器执行 top、df Exporter 自动暴露、Prometheus 定时拉取
存储 分散在文本/Excel/数据库 自带时序数据库 TSDB，高效压缩
查询 grep、awk 临时分析 PromQL 灵活多维查询与聚合
可视化 无 Grafana Dashboard 图表、模板、下钻
告警 邮件/短信脚本 Alertmanager 多路由、抑制、静默
扩展性 差 水平扩展、联邦集群、Remote Storage

一句话总结：没有监控的系统就像闭着眼睛开车，Prometheus + Grafana 让后端工程师从”救火”变成”预防”。

2、五大核心组件职责

组件 职责 典型场景
Prometheus Server 拉取指标、存储时序数据、执行 PromQL 查询、触发告警规则 核心大脑

3、Pull vs Push：为什么 Prometheus 选择 Pull

大多数监控系统（如 Zabbix、OpenTelemetry Collector）采用 Push 模型：被监控端主动把数据推送到中心。Prometheus 则采用 Pull 模型：服务端主动到目标拉取 /metrics。

维度 Pull 模型 Push 模型
控制方 Prometheus Server 决定何时拉、拉谁 客户端决定何时推
服务发现 天然支持 Kubernetes/Consul 自动发现 需要额外注册中心
网络要求 服务端能访问 Exporter 客户端能访问服务端
故障排查 直接 curl 目标看是否暴露指标 需要检查客户端推送链路
多副本 多个 Prometheus 可独立拉取同目标 容易造成重复推送
短任务 不适合，需要 Pushgateway 更适合

实战建议：长期运行的服务用 Pull；批处理/一次性任务用 Pushgateway 中转；防火墙限制下可改用 Pushgateway 或 Remote Write。

三、Prometheus 四种核心指标类型

Prometheus 的数据模型由 指标名（metric name）+ 标签（labels） 组成：http_requests_total{method="GET",status="200"}。理解四种指标类型是写 PromQL 和 exporter 的前提。

1、Counter（计数器）

只增不减，代表累计值，适合统计请求总数、错误总数、处理字节数等。

2、Gauge（仪表盘）

可增可减，代表瞬时值，适合 CPU 使用率、内存占用、当前连接数、队列长度等。

3、Histogram（直方图）

将采样值划分到多个桶（bucket）中，自动统计样本分布和总数、总和，适合接口耗时、响应大小等。

4、Summary（摘要）

客户端直接计算分位数，比 Histogram 更精确但无法聚合，适合单实例内部监控。

类型 是否可减 是否支持聚合分位 典型使用
Counter 否 否 请求总数、错误数、发送字节数
Gauge 是 否 CPU、内存、温度、当前连接
Histogram 否 可聚合 接口延迟分布、请求体大小
Summary 否 不可聚合 单进程精确分位、客户端预聚合

避坑提示：分位延迟优先用 Histogram，因为 Summary 的 quantile 是客户端预计算，跨实例聚合会产生误导。

四、Exporter 实战：从机器到业务应用

Exporter 是 Prometheus 生态的眼睛。常见的 Exporter 包括 Node Exporter（机器）、Blackbox Exporter（拨测）、MySQL Exporter、Redis Exporter、Kafka Exporter、JMX Exporter 等。

八、生产最佳实践与避坑清单

1、指标命名与标签规范

  • 指标名使用 snake_case，以 _total、_bytes、_seconds 等单位后缀结尾。
  • 标签值必须稳定，避免使用用户 ID、订单 ID、Trace ID 作为标签（高基数）。
  • 控制单条时间序列数量，Prometheus 官方建议单实例不超过百万级 series。

2、采集与存储规划

项目 建议
scrape_interval 生产 15s，关键业务可 5s，非关键 30s~1m
留存周期 本地 TSDB 7~15 天，长期存储用 Thanos/Cortex/VictoriaMetrics
高可用 多 Prometheus 副本 + Remote Write + Thanos Sidecar/Query
告警阈值 基于历史分位动态阈值，避免固定阈值导致误报

3、告警治理原则

  • 告警即行动：每条告警都应有明确的处理 SOP，否则不要发。
  • 分层路由：P0 电话/SMS，P1 企业微信，P2 邮件/钉钉。
  • 抑制与静默：维护窗口必须静默，级联故障通过 inhibit_rules 抑制非根因告警。
  • 告警模板：使用 {{ $labels }} 和 {{ $value }} 让值班人员一眼看到实例与数值。

4、常见坑点

坑点 现象 解决
高基数标签 Prometheus 内存暴涨、查询变慢 去掉高变动标签，或用 Recording Rule 预聚合
Counter 重置 服务重启后 rate() 出现尖峰 Prometheus 自动处理 Counter 重置，不用手动
histogram_quantile 跨实例聚合 分位数值失真 先 sum(rate(...)) by (le) 再计算分位
告警 for 时间过短 抖动导致频繁告警 关键指标至少 for: 5m
磁盘 IO 不足 TSDB 压缩/查询卡死 使用 SSD，监控 prometheus_tsdb_wal_corruptions_total

十、总结

Prometheus + Grafana 已经成为云原生时代监控体系的标配。掌握它的核心不在于会配几个 Exporter，而在于建立一套完整的可观测思维：指标要分层、告警要可行动、Dashboard 要服务于故障定位、监控本身也要有监控。配合 Alertmanager 的告警治理、Recording Rule 的查询优化、Remote Storage 的长期归档，才能真正把监控系统从”装饰品”变成生产稳定性保障的核心武器。

希望这篇文章能帮你把 Prometheus + Grafana 从”听说过”变成”能落地”。如果你正在搭建监控体系，建议先从 Node Exporter + Spring Boot Micrometer + Grafana 主机/应用大盘入手，再逐步引入中间件、告警和长期存储。监控这件事，永远不怕晚，就怕不开始。
