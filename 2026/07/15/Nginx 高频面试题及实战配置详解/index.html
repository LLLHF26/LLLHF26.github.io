Nginx 高频面试题及实战配置详解

一、基础概念篇（入门必问）

1、Nginx 是什么？核心定位是什么？

参考答案：

Nginx（engine x）是一款由俄罗斯程序员 Igor Sysoev 开发的轻量级、高性能 HTTP 和反向代理 Web 服务器，同时也是 IMAP/POP3/SMTP 代理服务器。全球 Top 1000 网站中超 40% 使用 Nginx，是互联网后端架构中最主流的 Web 服务器。

核心定位：

  • 高性能 Web 服务器：替代 Apache 处理静态资源，并发能力是 Apache 的 5~10 倍
  • 反向代理服务器：将客户端请求转发到后端真实应用服务器（Tomcat、Spring Boot、Node.js 等）
  • 负载均衡器：将请求分发到多台后端服务器，实现高可用、水平扩展
  • 动静分离服务器：处理静态资源（HTML/JS/CSS/图片），转发动态请求到后端
  • API 网关前置层：HTTPS 卸载、限流、防盗链、灰度发布、安全防护

生产环境最常见用途：Web 服务器 + 反向代理 + 负载均衡。

2、Nginx 和 Apache 的核心区别？为什么要选 Nginx？

参考答案：

对比维度 Nginx Apache
并发模型 异步非阻塞事件驱动（epoll） 同步阻塞多进程/多线程（select）
并发能力 高并发（几万~几十万） 低并发（几千），高并发下性能急剧下降
内存占用 极低（几MB~几十MB） 较高（一个连接一个线程）
静态资源处理 效率极高（异步IO） 效率一般
动态资源处理 需转发到后端（PHP-FPM/Tomcat等） 原生支持 mod_php 等模块
配置复杂度 简洁，集中式配置 复杂，支持 .htaccess 分布式配置
模块加载方式 编译时确定，动态模块支持有限 运行时动态加载模块（DSO）
适用场景 高并发互联网项目、反向代理、负载均衡 传统企业级应用、需大量动态模块项目

核心结论：高并发互联网项目首选 Nginx；传统企业级、对动态模块丰富度要求高的项目可选 Apache。面试回答时强调 Nginx 的”异步非阻塞事件驱动”模型，这是高并发的根本原因。

3、Nginx 采用什么并发模型？核心原理是什么？

参考答案：

Nginx 采用异步非阻塞事件驱动模型（基于 Linux 的 epoll、kqueue 等多路复用 IO），是其高并发的核心。

核心原理剖析：

  1. Master-Worker 多进程架构：

    • Master 进程：负责管理 Worker 进程、加载配置、平滑升级，不处理请求
    • Worker 进程：真正处理客户端请求，数量一般与 CPU 核数相同
    • 每个 Worker 独立处理请求，互不干扰，单个 Worker 可同时处理成千上万个连接

  2. 异步非阻塞 IO：

    • 一个 Worker 进程通过 epoll 同时监听多个连接的事件（连接建立、数据可读、数据可写）
    • 当某个连接有事件触发时才处理，否则 Worker 进程可继续处理其他连接
    • 一个 Worker 进程可并发处理上万连接，避免 Apache 一个连接一个线程的资源开销

  3. 协程机制（非强制）：Nginx 内部对部分业务实现协程化，进一步减少上下文切换

对比 Apache prefork 模式：Apache 一个进程只能处理一个连接，高并发下需创建大量进程/线程，内存、CPU 切换开销巨大，Nginx 完美规避此问题。

4、Nginx 有哪些核心进程？作用分别是什么？

参考答案：

Nginx 启动后会运行两类进程：

  1. Master 进程（主进程）

    • 数量：仅 1 个
    • 核心职责：管理 Worker 进程、加载配置文件、平滑升级/回滚、监听信号
    • 不处理任何客户端请求，纯管理角色

  2. Worker 进程（工作进程）

    • 数量：由配置 worker_processes 决定（一般设为 CPU 核数）
    • 核心职责：真正处理客户端请求、转发、反向代理、负载均衡
    • 各 Worker 进程相互独立、互不通信，通过共享内存共享数据

辅助进程（可选）：

  • Cache Manager：管理磁盘缓存（proxy_cache、fastcgi_cache）
  • Cache Loader：启动时将磁盘缓存加载到内存

5、什么是反向代理？Nginx 如何实现反向代理？

参考答案：

正向代理：代理客户端，替客户端访问目标服务器（如 VPN、科学上网代理），客户端知道目标服务器地址。

反向代理：代理服务器，替服务器接收客户端请求，将请求转发到后端真实服务器，客户端不知道后端真实服务器，只与反向代理交互。

6、Nginx 支持哪几种负载均衡策略？各有什么优缺点？

参考答案：

Nginx 通过 upstream 模块实现负载均衡，支持6 种核心策略：

① 轮询（Round Robin，默认）

  • 优点：实现简单，请求平均分配
  • 缺点：不考虑服务器性能差异、当前负载

② 加权轮询（Weighted Round Robin，推荐生产使用）

  • 优点：按权重分配请求，适配服务器性能差异，生产首选

③ IP Hash（会话保持）

  • 原理：根据客户端 IP 哈希固定分配到同一台后端
  • 适用：需要会话保持的业务（如未做 session 共享的传统 Web 应用）
  • 缺点：IP 分布不均时负载不均

④ 最少连接（Least Connections）

  • 原理：将请求分配给当前连接数最少的服务器
  • 适用：请求处理时间差异大的场景（如长连接、大文件上传）

⑤ URL Hash（需第三方模块）

  • 原理：相同 URL 固定分配到同一台后端
  • 适用：缓存服务器（CDN 边缘节点）

⑥ Fair（响应时间优先，需第三方模块）

  • 原理：按后端服务器响应时间分配，响应快的优先
  • 缺点：需安装 nginx-upstream-fair 模块

生产环境推荐：加权轮询（通用）+ IP Hash（会话保持）+ 最少连接（长连接场景）

7、什么是动静分离？Nginx 如何实现？

参考答案：

动静分离：将网站静态资源（HTML/JS/CSS/图片/视频）和动态资源（接口/JSP/PHP）分开处理，静态资源由 Nginx 直接处理，动态请求转发到后端应用服务器。是互联网项目提升性能的核心架构。

核心优势：

  • Nginx 处理静态资源能力是 Tomcat 的 10 倍以上，大幅降低后端压力
  • 静态资源可加浏览器缓存、CDN 加速、压缩，进一步提升访问速度
  • 后端专注处理动态业务，代码更易维护

8、Nginx 怎么处理请求的？location 匹配规则是什么？

参考答案：

Nginx 通过 location 指令匹配客户端请求 URL，决定请求由哪个规则处理。location 匹配是 Nginx 面试必问、配置最常踩坑的点。

location 匹配规则优先级（从高到低）：

匹配类型 语法 优先级 说明
精确匹配 location = /uri 最高 URL 完全等于 /uri 才匹配
前缀匹配（优先最长） location ^~ /uri 高 以 /uri 开头，且不再检查正则
正则匹配（区分大小写） location ~ pattern 中 区分大小写的正则匹配
正则匹配（不区分大小写） location ~* pattern 中 不区分大小写的正则匹配
普通前缀匹配 location /uri 低 以 /uri 开头的前缀匹配

匹配流程：

  1. 先判断精确匹配（=），命中则直接使用，停止匹配
  2. 记录最长普通前缀匹配（^~ 或普通前缀），暂存
  3. 按顺序匹配正则规则（~ / ~*），命中第一个即停止
  4. 若正则未命中，使用步骤 2 暂存的最长前缀
  5. 若无前缀匹配，返回 location / 通用规则

13、Nginx 是怎么处理 HTTP 请求的？底层流程是什么？

参考答案：

Nginx 处理 HTTP 请求的完整生命周期：

  1. 接收请求：Worker 进程通过 epoll 监听客户端连接，有新请求时触发读事件
  2. 解析请求行/请求头：Worker 解析 HTTP 请求方法、URL、协议版本、Header
  3. 匹配 Server/Location：根据 server_name 匹配虚拟主机，根据 URL 匹配 location 规则
  4. 执行 11 个阶段处理请求（Nginx 核心机制）：
    • NGX_HTTP_POST_READ_PHASE：realip 模块读取真实客户端 IP
    • NGX_HTTP_SERVER_REWRITE_PHASE：Server 级别 URL 重写
    • NGX_HTTP_FIND_CONFIG_PHASE：匹配 location
    • NGX_HTTP_REWRITE_PHASE：Location 级别 URL 重写
    • NGX_HTTP_POST_REWRITE_PHASE：重写后处理
    • NGX_HTTP_PREACCESS_PHASE：访问前预处理（限流、限连接）
    • NGX_HTTP_ACCESS_PHASE：访问权限校验（allow/deny）
    • NGX_HTTP_POST_ACCESS_PHASE：访问后处理
    • NGX_HTTP_PRECONTENT_PHASE：内容生成前（try_files）
    • NGX_HTTP_CONTENT_PHASE：内容生成阶段（核心，处理请求）
    • NGX_HTTP_LOG_PHASE：记录访问日志
  5. 生成响应：返回响应给客户端（静态文件 / 反向代理 / FastCGI）
  6. 日志记录：记录访问日志、错误日志

关键点：NGX_HTTP_CONTENT_PHASE 是最核心的阶段，所有实际处理请求的模块（static、proxy、fastcgi、uwsgi）都在此阶段生效。

五、生产踩坑与高频问题篇

17、Nginx 常见 502/504 错误原因及解决方案？

参考答案：

502 Bad Gateway：Nginx 作为反向代理，从上游服务器（Tomcat/PHP-FPM）接收到无效响应。

504 Gateway Timeout：Nginx 在规定时间内未收到上游服务器的响应。

错误码 常见原因 解决方案
502 上游服务宕机 检查上游服务进程是否存活、端口是否监听
502 上游端口配置错误 检查 proxy_pass / fastcgi_pass 端口与实际服务一致
502 上游响应数据过大 调大 proxy_buffer_size / proxy_buffers
502 php-fpm 配置错误 检查 fastcgi_pass 是 unix socket 还是 TCP
504 上游服务响应慢 调大 proxy_read_timeout / fastcgi_read_timeout
504 网络问题 检查 Nginx 与上游服务器网络连通性
504 上游服务高负载 优化上游服务性能、扩容

18、Nginx 为什么比 Apache 高并发？epoll 是什么？

参考答案：

核心原因：IO 多路复用模型不同。

传统 BIO（Apache prefork）：

  • 一个进程/线程处理一个连接
  • 高并发下需创建大量进程/线程
  • 进程/线程上下文切换开销巨大
  • 内存占用随连接数线性增长

NIO + IO 多路复用（Nginx）：

  • 一个进程通过 epoll 同时监听成千上万个连接
  • 只有当连接有事件触发时才处理，否则阻塞在 epoll
  • 无需为每个连接创建进程/线程，内存占用极低
  • 单机可轻松处理 5 万~10 万并发连接

epoll 是什么：

  • Linux 内核 2.6+ 提供的高性能 IO 多路复用机制
  • 核心优势：
    • 无最大连接数限制（select 受 FD_SETSIZE 1024 限制）
    • 时间复杂度 O(1)，与连接数无关
    • 使用红黑树 + 就绪链表管理 fd，效率极高
    • 边缘触发（ET）+ 水平触发（LT）模式，灵活高效

面试回答模板：Nginx 性能高的根本原因是”异步非阻塞 + epoll IO 多路复用”，单 Worker 进程可同时处理数万并发连接，避免了多线程模型的上下文切换开销。
